# Aurora M2 总体拆分与里程碑 V1.0

> 阶段名称：MVP认知闭环  
> 阶段权重：35%  
> 启动基线：Aurora V1.1 MVP对象契约

## 1. M2目标

把M1中已验证的静态对象链变成可重复运行的离线MVP：

```text
提交材料
→ 采集/导入
→ 解析与分段
→ Fact / Claim / Evidence提取
→ 知识卡片
→ 正反与限定观点
→ Insight
→ PersonalOpinion草案
→ Markdown简报
```

M2结束时，用户应能针对一个主题提交网页、PDF或音视频转写材料，并得到可追溯的研究简报。

## 2. 原则

1. 先离线确定性，再网络自动化；
2. 先可追溯，再追求智能；
3. 自动化输出必须通过V1.1校验；
4. Claim不能自动升级为Fact；
5. AI输出不能作为Evidence；
6. PersonalOpinion只生成Draft；
7. 每一步记录ProcessingRun；
8. 每个派生对象回溯到ContentUnit；
9. MVP继续使用SQLite；
10. 每个Issue保持端到端可运行。

## 3. 任务拆分

### M2-001：离线统一输入与内容单元化

```text
本地材料
→ Source
→ Document
→ ContentUnit
→ ProcessingRun
→ SQLite
```

支持Markdown、纯文本、结构化分段JSON、幂等导入、hash、CLI和失败重试。不包括网络、PDF解析、ASR和LLM。

### M2-002：多载体解析适配器

- 网页HTML；
- PDF文本；
- 视频/音频转写；
- 页面、段落、时间戳定位；
- Parser版本；
- 解析质量。

音视频优先导入已有转写，不自行训练ASR。

### M2-003：理解与证据提取

- Entity、DataPoint、Claim、Evidence、Fact候选；
- EvidenceRole；
- ClaimDimension；
- 原子性检查；
- 人工复核接口；
- 可替换Provider；
- Prompt版本和原始响应审计。

### M2-004：知识合成与观点草案

- KnowledgeObject；
- Relation；
- TimelineEntry；
- Insight；
- 支持/反驳/限定/上下文矩阵；
- PersonalOpinion Draft；
- 失效条件和复核时间。

### M2-005：工作流与Markdown输出

- 端到端Workflow；
- CLI或最小本地应用；
- Markdown研究简报；
- 重跑和错误恢复；
- 三个中芯国际案例回归；
- 扩展到至少20条黄金案例；
- M2 Exit Review。

## 4. 权重

| Issue | M2内部权重 | 完成标准 |
|---|---:|---|
| M2-001 | 15% | 离线导入稳定、幂等、可追溯 |
| M2-002 | 20% | 三类载体定位完整 |
| M2-003 | 25% | 提取结果可验证、可复核 |
| M2-004 | 20% | 知识到观点草案完整 |
| M2-005 | 20% | 端到端MVP与20条黄金集 |

## 5. 资源演进

M2-001不需要外部服务。

M2-002可能需要HTML/PDF解析库和已有转写文本。

M2-003可能需要一个LLM Provider、Prompt测试数据、Token和成本记录。引入付费API前必须由项目负责人确认。

M2-004/M2-005暂不需要向量数据库即可完成MVP。

## 6. M2退出标准

- [ ] 网页、PDF、转写文本进入同一对象链；
- [ ] 所有派生对象可追溯到Source；
- [ ] Fact/Claim边界测试通过；
- [ ] Evidence独立性生效；
- [ ] PersonalOpinion不自动激活；
- [ ] 可生成Markdown研究简报；
- [ ] 至少20条黄金测试案例；
- [ ] 全量测试通过；
- [ ] 总覆盖率≥90%；
- [ ] 无BLOCKER；
- [ ] MAJOR全部解决或明确延期；
- [ ] 婉儿QA通过；
- [ ] 项目负责人确认MVP可用。
