## 1. 定位

M2-003C 已经 CLOSED。OPT-073 是 M2 的收尾增强，聚焦在持久化边界的**健壮性加固**，不引入新功能、不修改核心映射逻辑、不改变已稳定的 API 契约。

**关键词**：边界补测 · 错误路径 · 并发特征化 · 风险边界 · 契约不可变

---

## 2. 目标

在 M2-003C 已交付的 7 个文件（contracts / identity / validation / mapper / draft_service / persistence_policy / draft_persistence）基础上：

1. **补全错误路径测试**：确保所有显式 `raise`、`return None`、`try/except` 路径有逐条测试；
2. **并发特征化与风险边界验证**：记录 SQLite 单写入模型下重复调用、冲突及失败审计的当前可观察行为，不将特征化结果提升为数据库级并发保证；
3. **追溯性审计**：以 [16 号 Gate 3 正式任务卡](16_M2-003C_Gate3_草案持久化验证_任务卡_V1.0.md) 为 G3-1～G3-7 的唯一规范来源，确认每条硬门禁都有**至少一条**直接对应的测试用例；
4. **冻结边界契约**：`contracts.py` 中所有公开函数签名不可变，并由测试守护。

---

## 3. 范围内

```
1. 错误路径补测
   ├── draft_service.py 中所有显式 raise 的路径 → 逐条测试
   ├── validation.py 中所有 return (False, reason) 的分支 → 逐条测试
   ├── mapper.py 中所有 _safe_enum → None 的枚举路径（已有部分，核查完整性）
   └── draft_persistence.py 中 draft_service 调用失败的传播路径

2. 并发特征化与风险边界验证（SQLite single-writer）
   ├── 同一 operation key / ProcessingRun 重复调用的当前可观察行为
   ├── 两个 Session 并发写入同一自然键时的 IntegrityError 或锁冲突行为
   └── 失败后是否形成稳定、可核验的失败审计

3. 门禁追溯矩阵
   ├── G3-1 ~ G3-7 每条 → 至少一个 test_ 函数
   ├── 唯一规范来源 → 16 号 Gate 3 正式任务卡
   ├── G3-7 → Provider 控制或越权语义直接进入持久化核心对象的数量必须为 0
   └── 缺失的直接补充

4. 契约冻结测试
   ├── contracts.py 公开函数签名测试（参数名/类型/返回值类型）
   └── identity.py 确定性哈希不变性测试
```
