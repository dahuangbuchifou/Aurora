# M1-003A：Aurora V1.1 变更清单与评审结论 V1.0

> **阶段**：M1-003A  
> **目标**：评审真实案例暴露的问题，冻结 V1.1 设计  
> **原则**：不在本阶段修改代码，不把 WQRS 专属逻辑写入 Aurora 通用模型

---

M1-001 完成了 17 种核心对象及持久化层。

```text

```

目标服务器 QA 结果：


M1-002 证明了对象模型的主干成立，同时暴露出财务数据口径、证据独立性、

## 3. 逐项评审结论

### OPT-001：DataPoint 测量与财务口径

**结论：入选 M1-003B，Schema 变更。**

新增可选嵌套对象：

```text
measurement_context: MeasurementContext
```

入选原因：

- 当前 `value + unit + period` 无法区分人民币、美元、元、千元和亿元；
- 无法表达 CAS、IFRS 等报告标准；
- 无法区分归母、公司整体、业务分部等口径；
- 问题已由真实年报案例证明；
- 使用嵌套对象可以避免把金融字段散落在 DataPoint 顶层。

通用性约束：

- 字段命名使用 `reporting_standard`，不限定为会计准则；
- 财务字段全部可选；
- 非金融 DataPoint 不承担额外必填负担；
- WQRS 的估值、财务分析规则仍留在 WQRS。

不入选字段：

```text
value_type
```

原因：与现有 `calculation_method`、`unit` 部分重叠。

替代为语义更清晰的：

```text
measurement_kind
```

---

### OPT-002：Evidence 独立性

**结论：入选 M1-003B，行为变更，不新增核心字段。**

现有：

```text
independence_group
```

已经足够表达来源独立组，缺失的是统一计算规则。

冻结规则：

```text
独立证据键 =
(target_object_id, evidence_role, independence_group)
```

同一个键下可以保留多个 Evidence 和 Document，但在“独立证据数量”统计中只计一次。

入选原因：

- Case A 官网 PR 和 Case C 年报属于同一原始披露；
- 去重应发生在聚合层，不应删除来源；
- 无需再增加一个重复字段。

M1-003B 需要提供：

```text
group_independent_evidence()
count_independent_evidence()
validate_independence_groups()
```

---

### OPT-003：Claim 分析维度

**结论：入选 M1-003B，Schema 变更。**

新增：

```text
claim_dimension: ClaimDimension = general
```

冻结语义：

- `claim_type`：说话者在做什么类型的主张；
- `claim_dimension`：该主张讨论哪个分析维度。

示例：

```text
“2026年收入将增长”
claim_type      = prediction
claim_dimension = business_growth

“170元以上不买”
claim_type      = recommendation
claim_dimension = valuation
```

使用单值而不是数组，原因：

- 一条 Claim 应有一个主要分析维度；
- 有助于推动 Claim 原子化；
- 多个维度应优先拆成多个 Claim；
- 避免维度标签失控。

---

### OPT-004：Provenance 派生关系

**结论：入选 M1-003B，Schema 变更。**

新增：

```text
derivation_links: list[DerivationLink]
```

每个 Link 包含：

```text
object_id
relation_type
note
```

原有：

```text
origin_object_ids
```

V1.1 保留，不删除、不重命名。

入选原因：

- `origin_object_ids` 只能表达“来自”，不能表达如何派生；
- PR 摘要年报、翻译、计算和 AI 推断的语义不同；
- 派生方式会影响证据独立性和可信度。

V1.0 适配时，将旧 `origin_object_ids` 映射为：

```text
relation_type = derived_from
```

不猜测更具体的关系。

---

### OPT-005：Claim 原子性

**结论：部分入选。**

进入 M1-003B：

- 原子 Claim 规范；
- 黄金案例；
- 警告式 Linter；
- 质量标记；
- 拆分建议。

不进入 M1-003B：

- 基于关键词的硬性 Pydantic 拦截；
- 自动拆分复合 Claim；
- 新增复杂的 ClaimGroup 核心对象。

原因：

- 语义原子性无法通过稳定的静态规则完全判断；
- 硬拦截容易产生大量误报；
- 自动拆分属于 Understanding/LLM 阶段；
- 当前应先建立可审计规范。

冻结原则：

> 一条 Claim 只表达一个可被支持、反驳或验证的主要判断。

Linter 只返回：

```text
INFO / WARNING
```

不得直接阻止存储。

---

### OPT-006：SQLite busy timeout

**结论：入选 M1-003B，工程变更。**

新增配置：

```text
AURORA_SQLITE_TIMEOUT_SECONDS=30
```

SQLite 连接设置：

```text
connect_args.timeout
PRAGMA busy_timeout
```

原因：

- Linux 裸机和后续工作流可能出现短暂写锁；
- 统一配置比在 `session_scope` 中增加 timeout 更合理；
- 不改变 Repository 和 Session 的职责。

---

### OPT-007：测试补强

**结论：入选 M1-003B。**

必须补充：

- schema_registry 缺少、未知和不支持版本；
- V1.0/V1.1 混合读取；
- session_scope commit、rollback、close；
- SQLite timeout 配置；
- migration dry-run、备份、恢复；
- 非依赖引用的 dangling 行为。

原因：

- 属于 V1.1 可靠性基线；
- 成本低、价值明确；
- 不扩大业务范围。

### OPT-014：Evidence Role 判定标准

**结论：入选 M1-003B，语义与测试变更，不新增字段。**

冻结定义：

| Role | 含义 |
|---|---|
| support | 在相同主体、时间和范围下，提高目标主张的可信度 |
| refute | 在相同主体、时间和范围下，直接否定目标主张 |
| qualify | 缩小适用范围、增加条件或降低确定性，但不直接否定 |
| context | 提供背景，不对主张方向产生明确支持或反驳 |
| counterexample | 提供具体反例，挑战一般性或普遍性 |

关键规则：

```text
增长预期
与
估值偏贵
```

通常属于不同维度，不能自动标为 `refute`。

---

### OPT-015：V1.0 → V1.1 兼容

**结论：入选 M1-003B。**

采用：

```text
加法式升级
双版本读取
读取时适配
混合版本存储
显式离线迁移
```

禁止：

- 读取旧对象时自动写回；
- 在 Alembic 中直接批量改 JSON payload；
- 删除 V1.0 Schema；
- 把缺失新字段默认为真实业务信息。

---

### OPT-016：processing_run_id 追溯噪音

**结论：入选 M1-003B，Bugfix。**

冻结规则：

- `provenance.processing_run_id` 是审计引用；
- Traceability Edge 标记 `dependency=false`；
- 缺失的非依赖目标不计入 dangling error；
- 可以记录为 advisory warning；
- 如果 ProcessingRun 存在，仍允许追溯；
- 认知链完整性不得依赖 ProcessingRun 必须随案例导出。

## 5. 明确非目标

M1-003B 不做：

- 自动 Claim 拆分；
- LLM 抽取；
- 网页/PDF/视频自动解析；
- WQRS 财务规则；
- 估值和买卖建议；
- 全文或向量检索；
- 多用户权限；
- 对象表全面规范化；
- 20 条黄金测试集扩建。
