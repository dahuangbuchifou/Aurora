# M1-003A：ADR 与 V1.1 字段定义 V1.0

> 本文件记录 Aurora V1.1 的不可轻易回退设计决策。

## ADR-005：DataPoint 使用可选 MeasurementContext

### 状态

```text
PROPOSED_FOR_ACCEPTANCE
```

### 决定

在 `DataPoint` 新增：

```python
measurement_context: MeasurementContext = Field(
    default_factory=MeasurementContext
)
```

定义：

```python
class MeasurementKind(StrEnum):
    UNKNOWN = "unknown"
    MONETARY = "monetary"
    RATIO = "ratio"
    PERCENTAGE = "percentage"
    COUNT = "count"
    PHYSICAL = "physical"
    INDEX = "index"
    RATE = "rate"
    OTHER = "other"


class MeasurementContext(AuroraModel):
    measurement_kind: MeasurementKind = MeasurementKind.UNKNOWN
    currency: str | None = None
    scale_multiplier: int = 1
    reporting_standard: str | None = None
    attribution_scope: str | None = None
    consolidation_scope: str | None = None
```

### 字段语义

| 字段 | 说明 | 示例 |
|---|---|---|
| measurement_kind | 数据性质 | monetary、percentage |
| currency | ISO 4217 三位币种代码 | CNY、USD |
| scale_multiplier | 原值乘数 | 1、1000、100000000 |
| reporting_standard | 报告或计量标准 | CAS、IFRS |
| attribution_scope | 数值归属范围 | attributable_to_parent |
| consolidation_scope | 合并范围 | consolidated、parent_only、segment |

### 校验

- `scale_multiplier > 0`；
- currency 非空时必须满足大写三位字母；
- `measurement_kind=monetary` 且 currency 缺失时返回质量警告；
- V1.0 适配对象默认为 `measurement_kind=unknown`；
- 不根据 `unit` 自动猜测会计口径。

### 兼容性

完全加法式，V1.0 对象不需要提供该字段。

### 不采用的方案

将所有字段直接放入 DataPoint 顶层。

原因：

- 污染通用对象；
- 非金融数据出现大量空字段；
- 后续难以扩展测量语境。

## ADR-006：ClaimType 与 ClaimDimension 正交

### 状态

```text
PROPOSED_FOR_ACCEPTANCE
```

### 决定

新增：

```python
class ClaimDimension(StrEnum):
    GENERAL = "general"
    BUSINESS_GROWTH = "business_growth"
    FINANCIAL_PERFORMANCE = "financial_performance"
    VALUATION = "valuation"
    RISK = "risk"
    MARKET_EXPECTATION = "market_expectation"
    ACTION_RECOMMENDATION = "action_recommendation"
    POLICY = "policy"
    TECHNOLOGY = "technology"
    OPERATIONS = "operations"
    GOVERNANCE = "governance"
    SUPPLY_CHAIN = "supply_chain"
    OTHER = "other"
```

Claim 新增：

```python
claim_dimension: ClaimDimension = ClaimDimension.GENERAL
```

### 语义边界

| 字段 | 回答的问题 |
|---|---|
| claim_type | 这是事实性断言、解释、预测、建议还是价值判断？ |
| claim_dimension | 该断言主要讨论增长、估值、风险还是技术？ |

### 决策理由

- 同一 `prediction` 可讨论增长、风险或政策；
- 同一 `valuation` 维度可表现为解释、判断或建议；
- 单值维度促进 Claim 原子化；
- 不使用任意字符串，防止分类漂移。

### V1.0 适配

缺失时填入：

```text
general
```

不得根据 statement 自动猜测。

## ADR-007：Provenance 增加结构化 DerivationLink

### 状态

```text
PROPOSED_FOR_ACCEPTANCE
```

### 决定

新增：

```python
class DerivationRelationType(StrEnum):
    DERIVED_FROM = "derived_from"
    QUOTED_FROM = "quoted_from"
    SUMMARIZES = "summarizes"
    REPOSTS = "reposts"
    CALCULATED_FROM = "calculated_from"
    TRANSLATED_FROM = "translated_from"
    EXTRACTED_FROM = "extracted_from"
    INFERRED_FROM = "inferred_from"


class DerivationLink(AuroraModel):
    object_id: str
    relation_type: DerivationRelationType
    note: str | None = None
```

Provenance 新增：

```python
derivation_links: list[DerivationLink] = Field(default_factory=list)
```

### 兼容规则

原字段继续保留：

```python
origin_object_ids: list[str]
```

V1.1 验证：

```text
origin_type=derived
时
origin_object_ids 或 derivation_links 至少一个非空
```

读取 V1.0 时：

```text
origin_object_ids
→ derivation_links(relation_type=derived_from)
```

该转换只发生在内存适配层，不自动写回数据库。

## ADR-008：Evidence 独立性在聚合层实现

### 状态

```text
ACCEPTED_DESIGN
```

### 决定

不新增 Evidence 字段。

独立证据聚合键：

```python
(
    evidence.target_object_id,
    evidence.evidence_role,
    evidence.independence_group,
)
```

### 行为

- 同组多个 Evidence 全部保留；
- 独立证据计数只计一次；
- 支持和反驳角色分开计数；
- 不跨 target 合并；
- 不因去重删除 Document、ContentUnit 或 Evidence。

### 原因

`independence_group` 已能表达同源关系，问题在算法而不是 Schema。

## ADR-009：Claim 原子性采用规范与 Linter

### 状态

```text
ACCEPTED_WITH_LIMITATION
```

### 决定

M1-003B 不加入硬性 Pydantic 语义校验。

新增警告式检查：

```text
ClaimAtomicityLinter
```

输出：

```text
INFO
WARNING
```

可能检查：

- 一个 statement 同时出现多个独立结论；
- 同时混入预测和行动建议；
- 同时包含互不依赖的数值；
- 条件和结论边界不清；
- statement 过长。

### 限制

这些检查不能证明 Claim 一定是复合的，只能提示人工复核。

## ADR-010：EvidenceRole 采用同范围直接性原则

### 状态

```text
ACCEPTED_DESIGN
```

### 决定

Evidence Role 判定前，先比较：

```text
主体
时间
口径
分析维度
适用条件
```

只有在主要范围一致时，才可标记：

```text
support / refute
```

否则优先考虑：

```text
qualify / context
```

`counterexample` 用于具体反例挑战普遍性。

### 典型示例

```text
管理层：“收入会增长”
UP主：“当前估值偏贵”
```

不是直接反驳，通常是 `qualify` 或并列的不同维度观点。

## ADR-011：非依赖审计引用不参与悬空错误

### 状态

```text
ACCEPTED_DESIGN
```

### 决定

Traceability Edge 必须包含：

```python
dependency: bool
```

验证规则：

```text
dependency=true 且目标缺失
→ dangling error

dependency=false 且目标缺失
→ advisory warning 或忽略
```

以下默认是非依赖引用：

```text
provenance.processing_run_id
```

以下默认是依赖引用：

```text
source_id
document_id
source_ref
origin_object_ids
derivation_links.object_id
supporting_object_ids
evidence_ids
knowledge_refs
insight_refs
opinion_refs
```

## ADR-012：V1.1 采用双版本读取与混合存储

### 状态

```text
PROPOSED_FOR_ACCEPTANCE
```

### 决定

- 保留 V1.0 Schema；
- 新增 V1.1 Schema；
- Repository 可读取两种版本；
- V1.0 在内存中适配为当前模型；
- 读取行为不自动持久化；
- 新对象和显式更新对象写 V1.1；
- 数据库允许混合版本；
- 批量升级必须显式执行。

Schema 目录：

```text
schemas/
├── v1/          # 现有V1.0，冻结
├── v1_1/        # 新V1.1
└── registry.json
```

## ADR-013：JSON payload 迁移不由 Alembic 隐式执行

### 状态

```text
PROPOSED_FOR_ACCEPTANCE
```

### 决定

Alembic 只负责 DDL。

M1-003B 的 Alembic Revision：

```text
新增 object_records.schema_version 索引
```

JSON payload 升级通过独立命令：

```text
aurora-migrate-payloads
```

原因：

- Payload 是领域数据，不是单纯表结构；
- 需要 dry-run、逐对象校验和备份；
- 不应在 `alembic upgrade` 中静默改写全部知识资产；
- 失败时需要按对象恢复。

## ADR-014：SQLite 超时配置属于 Engine

### 状态

```text
ACCEPTED_DESIGN
```

### 决定

环境变量：

```text
AURORA_SQLITE_TIMEOUT_SECONDS=30
```

配置位置：

```text
create_db_engine()
```

不得向：

```text
session_scope()
```

添加不明确的 timeout 参数。
