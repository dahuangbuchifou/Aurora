## 七、核心设计说明

### 1. 为什么使用单表 JSON 持久化

M1 是对象模型验证阶段。当前采用：

```text
公共元数据列 + 完整 JSON payload
```

这样可以在不提前固化 17 张业务表的情况下，验证对象能否表达真实资料。

索引字段：

- `id`
- `object_type`
- `workspace_id`
- `lifecycle_status`
- `updated_at`

后续根据真实查询负载逐步规范化，而不是一次性过度设计。

### 2. Repository 边界

当前 Repository 支持：

- create
- get / get_required
- list
- update
- soft_delete
- restore
- count
- record_version

当前不包含：

- 复杂跨对象查询；
- 关系图谱遍历；
- 全文搜索；
- 向量检索；
- API 层；
- Collector / Parser；
- LLM 调用。

### 3. 重要业务校验

- Prediction Claim 必须包含时间范围；
- Fact 必须包含 Evidence；
- 洞察必须有支持对象和推理步骤；
- Active PersonalOpinion 必须由用户确认；
- Active PersonalOpinion 必须包含支持、假设、失效条件和复核时间；
- Derived Provenance 必须指向来源对象；
- 非假设 Relation 必须包含证据；
- OutputArtifact 必须包含内容和知识引用；
- ProcessingRun 终态必须有完成时间；
- 软删除不会物理删除历史对象。

## 九、婉儿 QA 重点

建议重点检查：

1. 17 个对象是否符合 M0 Spec；
2. `Claim / Fact / Evidence` 边界；
3. `Insight / PersonalOpinion` 边界；
4. Active Opinion 的人工确认门槛是否合理；
5. JSON 单表持久化是否接受；
6. Schema 是否能覆盖后续三个真实案例；
7. SQLite 3.26 下 Migration 是否正常；
8. 完整测试是否通过。

## 十、已知限制

- 尚未对 3 个真实业务案例进行对象映射；
- 尚未建立 API；
- 尚未支持对象间数据库外键；
- 尚未支持全文和语义检索；
- 尚未规范化 Entity、Claim、Evidence、Relation；
- 尚未进行多进程并发压力测试；
- SQLite 时间字段自身不保留完整时区语义，权威值仍保存在 JSON payload 中；
- `model_copy(update=...)` 是 Pydantic 的低层方法，业务代码应优先通过重新校验或 Repository 更新。
