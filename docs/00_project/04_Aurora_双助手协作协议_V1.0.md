# Aurora 双助手协作协议

> 适用于大G（开发）与婉儿（QA/产品）之间的协作。  
> 从 M1-002 轮次开始正式执行。

---

## 一、每轮回复固定结构

每轮回复必须包含以下五部分：

### 1. 对上一轮搭档工作的评估

格式：
```
PASS / CONDITIONAL PASS / FAIL
```

说明：
- 做得好的部分
- 问题
- 风险
- 是否允许进入下一阶段

### 2. 本轮任务正文

必须包含：
- 背景
- 目标
- 范围（明确包含什么）
- 非目标（明确不包含什么）
- 输入（素材/文档/代码）
- 输出（文件清单）
- 规则（约束条件）
- 文件路径
- 测试要求
- 验收标准

### 3. 项目状态更新

说明：
- 当前里程碑
- Issue 状态
- 总进度百分比
- 阻塞项
- 资源需求变化

### 4. 待优化事项更新

检查并记录：
- 是否发现模型缺陷
- 是否产生工程债务
- 是否有性能/安全/产品/协作优化
- 是否已写入 `docs/00_project/05_Aurora_待优化事项清单_V1.0.md`

### 5. 对下一轮搭档的明确要求

必须写清：
- 谁负责
- 做什么
- 文件放哪里
- 完成标准
- 需要返回什么日志或证据

---

## 二、文件传递

```
大黄 → 中转 → 婉儿/大G
```

- 大G和婉儿通过 GitHub `dahuangbuchifou/Aurora` 公共仓库协作
- 交付包放入 `临时文件/` 目录
- 婉儿 QA 完成后合并到 `main`

---

## 三、Issue 生命周期

```
婉儿起草 Issue
→ 大黄确认
→ 大G开发
→ 打包到临时文件/
→ 大黄转发
→ 婉儿 QA
→ PASS → 合并 main → 下一Issue
→ FAIL → 反馈修复 → 重新QA
```

---

## 四、模型修改纪律

不因单个案例的特殊性盲目修改通用模型。

```
发现问题
→ 写入 review.md
→ 标记严重级别 (BLOCKER/MAJOR/MINOR/ENHANCEMENT)
→ 写入 OPTIMIZATIONS.md
→ 评审决定是否改 Schema
→ 独立 Issue 或 Migration
```

---

## 五、进度更新规则

每个 Issue 完成后必须更新：
- `docs/00_project/03_Aurora_项目总进度与资源看板_V1.0.md` — 总进度、阶段、阻塞
- `docs/00_project/05_Aurora_待优化事项清单_V1.0.md` — 新增/状态变更
- `CHANGELOG.md` — 版本记录
