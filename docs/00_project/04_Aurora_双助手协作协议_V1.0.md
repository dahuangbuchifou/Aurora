# Aurora 双助手协作协议 V1.0

> 角色：项目负责人、婉儿、大G  
> 目标：让需求、实现、测试、评估、优化和下一轮任务形成闭环。

## 1. 分工

### 婉儿：产品经理 + QA + Git执行

- 需求拆解、Spec、Issue
- 用户场景、验收标准、测试用例
- 服务器运行、Git分支、Commit、PR
- QA结论、产品缺陷和优化建议

### 大G：技术架构 + 编码 + 技术审查

- 架构、模块和接口边界
- 完整代码、DB Schema、Migration、API
- 自动化测试、技术风险、Bug修复
- 架构审查和兼容性设计

### 项目负责人

- 总体方向、范围、优先级和重大取舍
- 文件与信息中转
- 阶段发布确认

## 2. 每轮固定包含五部分

### A. 对上一轮搭档工作的评估

```markdown
## 上一轮工作评估
- 结论：PASS / CONDITIONAL PASS / FAIL
- 做得好的部分：
- 发现的问题：
- 风险等级：
- 是否允许进入下一阶段：
```

### B. 本轮任务正文

必须有：

- 背景、目标、范围、非目标
- 输入、输出、规则、文件路径
- 测试和验收标准

### C. 项目状态更新

必须说明：

- 当前里程碑和Issue状态
- 总体进度是否变化
- 阻塞项和资源需求

### D. 待优化事项更新

必须检查并记录：

- 模型缺陷
- 工程债务
- 性能、安全、易用性和协作优化

### E. 对下一轮搭档的明确要求

必须说明：

- 谁负责
- 交付物
- 路径
- 完成标准
- 必须返回的日志或证据

## 3. 标准流程

```text
婉儿创建Issue/Spec
→ 大G评审并实现
→ 项目负责人中转文件
→ 婉儿落库、测试和QA
→ 大G评估与修复
→ 婉儿回归
→ 项目负责人确认合并/切换阶段
→ 更新进度看板和优化清单
```

## 4. 大G交付模板

```markdown
# 任务交付
## 1. 上一轮工作评估
## 2. 本轮实现摘要
## 3. 变更文件树
## 4. 完整文件或交付包
## 5. 安装、迁移和测试命令
## 6. 验收标准映射
## 7. 已知限制
## 8. 待优化事项更新
## 9. 对婉儿下一轮要求
## 10. 项目进度与资源更新
```

## 5. 婉儿QA模板

```markdown
# QA报告
## 1. 对大G上一轮交付的评估
## 2. 测试环境和执行命令
## 3. 测试结果
## 4. 验收标准逐项结果
## 5. Bug与产品问题
## 6. 待优化事项更新
## 7. 对大G下一轮要求
## 8. 项目进度与资源更新
```

## 6. Issue状态

```text
DRAFT
READY
IN_PROGRESS
READY_FOR_QA
QA_FAILED
QA_PASSED
BLOCKED
MERGED
CLOSED
```

## 7. 必须升级给项目负责人的事项

- MVP范围变化
- 核心对象删除或语义改变
- 破坏性Schema修改
- 不可逆迁移
- 新增付费基础设施
- 云端处理敏感数据
- QA为FAIL仍希望合并
