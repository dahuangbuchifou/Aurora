# M1-003A：Aurora V1.0 → V1.1 兼容矩阵 V1.0

---

## 1. 兼容目标

V1.1 必须满足：

- V1.0 数据可读取；
- V1.0 数据不因读取被自动改写；
- V1.1 数据可保存；
- 数据库允许 V1.0 与 V1.1 共存；
- 三个 M1-002 案例无需丢失信息；
- 不删除旧 Schema；
- 迁移失败可恢复。

---

## 2. 字段兼容矩阵

| 对象 | V1.0 | V1.1 | 读取旧数据 | 写入策略 | 破坏性 |
|---|---|---|---|---|---|
| DataPoint | 无 measurement_context | 新增可选对象 | 填默认 UNKNOWN | 新建写V1.1 | 否 |
| Claim | 无 claim_dimension | 默认 general | 填 general | 新建写V1.1 | 否 |
| Provenance | origin_object_ids | 增加 derivation_links | 映射为 derived_from | 新建优先写links | 否 |
| Evidence | independence_group | 字段不变 | 原样读取 | 聚合逻辑变化 | 否 |
| BaseObject | schema_version=1.0 | schema_version=1.1 | Adapter升级 | 新建写1.1 | 否 |
| Traceability | 非依赖仍报dangling | 非依赖不报错 | 无数据变更 | 行为修复 | 否 |

---

## 3. 枚举兼容矩阵

### 新增枚举

```text
MeasurementKind
ClaimDimension
DerivationRelationType
```

这些枚举没有替换 V1.0 枚举。

### 保持不变

```text
ClaimType
EvidenceRole
EvidenceType
CalculationMethod
```

### 原则

- V1.1 不重命名现有枚举值；
- 不改变已有枚举值含义；
- 新枚举必须有向后兼容默认值；
- 适配器不得根据文本内容自动猜测枚举。

---

## 4. 读取流程

```text
读取 object_records.payload
        ↓
检查 schema_version
        ↓
1.1 → 直接校验
1.0 → upgrade_payload_v1_0_to_v1_1()
        ↓
V1.1 Pydantic校验
        ↓
返回当前领域对象
```

要求：

- Adapter 使用 payload 副本；
- 不修改 SQLAlchemy Record；
- 不自动 commit；
- 适配失败必须返回明确错误；
- 保留 object id 和 created_at；
- 不制造业务信息。

---

## 5. V1.0 默认适配值

### DataPoint

```json
{
  "measurement_context": {
    "measurement_kind": "unknown",
    "currency": null,
    "scale_multiplier": 1,
    "reporting_standard": null,
    "attribution_scope": null,
    "consolidation_scope": null
  }
}
```

### Claim

```json
{
  "claim_dimension": "general"
}
```

### Provenance

当存在：

```json
{
  "origin_object_ids": ["cu_001"]
}
```

内存中补充：

```json
{
  "derivation_links": [
    {
      "object_id": "cu_001",
      "relation_type": "derived_from",
      "note": "legacy V1.0 compatibility mapping"
    }
  ]
}
```

不得猜测为：

```text
summarizes
quoted_from
calculated_from
```

## 6. 写入矩阵

| 场景 | 写入版本 |
|---|---|
| 新建对象 | 1.1 |
| 读取V1.0但未修改 | 不写回 |
| 显式更新V1.0对象 | 完整校验后写1.1 |
| 导入V1.0对象 | 可保持1.0或显式升级 |
| 离线批量迁移 | 写1.1并生成备份 |
| Migration失败 | 原对象不变或从备份恢复 |

## 7. Schema 目录兼容

```text
schemas/v1/
```

作为 V1.0 冻结目录，禁止被 V1.1 导出覆盖。

新增：

```text
schemas/v1_1/
```

顶层新增：

```json
{
  "latest": "1.1",
  "versions": {
    "1.0": "v1/registry.json",
    "1.1": "v1_1/registry.json"
  }
}
```

---

## 8. Repository 兼容要求

Repository 必须：

- 根据 payload schema_version 路由 Adapter；
- `get()` 返回当前领域模型；
- `create()` 只接受支持版本；
- `update()` 使用当前 V1.1 规则；
- `list()` 可返回混合来源版本的当前模型；
- 不因 list/get 自动更新数据库；
- 提供原始 payload 读取接口供审计和迁移使用。

建议新增：

```text
get_raw_record()
list_by_schema_version()
```

这两个接口只服务迁移、审计和运维，不暴露给普通业务调用。

---

## 9. 不兼容情况

以下输入应明确失败：

- schema_version 缺失；
- schema_version 不在支持列表；
- V1.0 数据本身已违反 V1.0 必填规则；
- 新增字段值不符合 V1.1 枚举；
- migration 后对象无法通过 V1.1 校验；
- downgrade 会丢失无法转换的 V1.1 信息。

禁止静默：

- 删除未知字段；
- 将未知币种猜为 CNY；
- 将 general Claim 猜成 valuation；
- 将 derived_from 猜成 summarizes；
- 因适配失败而返回部分对象。
