# M1-002 对象模型真实案例审计报告

## 审计结论

**CONDITIONAL PASS（模型可用，建议进入V1.1评审）**

M1-002没有发现阻止完整认知链运行的BLOCKER。三个案例可以创建、校验、持久化、查询和回溯。但若Aurora要进入真实财务研究和WQRS输入阶段，以下MAJOR问题必须在M1-003讨论。

## BLOCKER

无。

## MAJOR

### M-01 DataPoint 财务口径不足

缺少币种、数量级、会计准则、归属范围和合并范围。当前复合unit只能临时表达，容易造成同名指标不可比较。

### M-02 证据独立性只有标签，没有约束

`independence_group` 能标识Case A与Case C同源，但系统不会自动禁止评分模块把同一组Evidence重复计数。需要独立的聚合和计数规则。

### M-03 Claim 缺少分析维度

增长预测、估值判断、风险判断和行动建议可能同时成立。当前只有ClaimType，不能完整表达“不同维度、并非直接冲突”。建议评审 `claim_dimension` 或论证轴模型。

### M-04 Provenance 派生关系过于粗粒度

`origin_type=derived` 无法结构化区分 `summarizes`、`reposts`、`calculated_from`、`translated_from`。Relation可以补充，但两套表达尚未形成统一规范。

### M-05 复合Claim缺少原子性约束

Case A C03同时包含风险判断和可验证数值，且数值与Case C不一致。模型能通过`disputed + refute evidence`表达冲突，但没有强制一句话拆为单一可验证主张。

## MINOR

### m-01 Source引用字段重复

BaseObject的 `source_refs` 与 Claim/DataPoint/Evidence的 `source_ref` 可能造成两个来源入口不一致。

### m-02 asserted_by 类型不严格

Claim.asserted_by是字符串，既可放实体ID也可放普通名称，难以强制观点主体可追溯。

### m-03 时点数据表达不自然

价格和PE使用TimeRange.period表达，建议增加observed_at。

### m-04 Fact 与 DataPoint 存在重复维护风险

一个财务数值同时存在DataPoint和Fact。后续需明确Fact是“事实陈述”，DataPoint是“数值记录”，并规定更新同步方式。

## ENHANCEMENT

- 支持数值区间和置信区间；
- 保存原始展示值和标准化值；
- 为观点矩阵增加分析维度和兼容关系；
- 增加Evidence Group聚合器；
- 为财务数据增加审计报告和报表层级引用。

## 字段冗余与风险

- `metadata` 灵活但可能绕开Schema，本案例没有使用metadata承载关键字段；
- OutputArtifact同时支持content和content_uri，应规定正式输出的存储优先级；
- 多个对象各自拥有as_of/review字段，后续需统一时间治理规范。
