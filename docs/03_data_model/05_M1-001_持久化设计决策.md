# M1-001 持久化设计决策

## 决策

M1 阶段使用单表 `object_records` 保存全部 Aurora 对象：

- 稳定、高频查询的公共元数据使用独立列；
- 完整、经 Pydantic 校验的对象保存于 JSON `payload`；
- 使用 `object_type`、`workspace_id`、`lifecycle_status` 和 `updated_at` 建立索引；
- 使用 `version` 实现基础乐观并发控制。

## 原因

核心对象模型仍处于真实案例验证期。此时为 17 类对象分别建立高度规范化的数据表，会显著增加迁移成本，并可能把未经验证的字段结构固化。

当前方案优先保证：

1. Schema 可验证；
2. 对象可完整往返；
3. 版本可管理；
4. 数据库可迁移；
5. 后续可以逐步将高频字段规范化。

## 边界

本方案不是最终数据库设计。

进入稳定版后，应根据真实查询负载决定是否规范化：

- 实体；
- 事件；
- Claim；
- Evidence；
- Relation；
- 时间线；
- 全文索引和向量索引。

任何规范化升级必须通过 Alembic，并保持 Pydantic 对象契约向后兼容。
