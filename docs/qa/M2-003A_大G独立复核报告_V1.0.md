> 复核对象：Case A/B/C expected results、Expected Results Schema、Gate 0 Checker、M2-003A Issue  
> 复核基线：GitHub main 当前版本  
> 结论日期：2026-07-12

## 1. 总体裁决

```text
交付完整性：PASS
机械结构预检查：PASS
独立语义复核：FAIL
Gate 0：NOT_PASSED

```

`21/21 passed` 只能说明三个文件分别通过了七项弱结构预检查，不能证明 Gate 0 的语义目标已经通过。

当前需要修订三份 Expected Results、Schema 和 Checker，然后由婉儿复核、大黄裁断争议项。

## 2. 阻塞性争议

### D-001：Case A 代码片段被放入 expected_claims

当前：

```text
source = "annual_report"
→ expected_claims
→ claim_type=fact_claim
```

但该条目的 note 同时说明它不是事实主张。

裁决建议：

```text
从 expected_claims 删除
→ 移入 expected_rejects
→ reason=CONTENT_UNIT_CODE_BLOCK_NON_CLAIM
```

黄金集不能把“应拒绝对象”同时定义为“预期 Claim”。

### D-002：Case A 未来意图的 ClaimType

原文：

```text
管理层表示将继续推进产能建设
```

当前标为：

```text
fact_claim
```

建议改为：

```text
prediction
statement=公司将继续推进产能建设
asserted_by=中芯国际
time_horizon={precision: unknown}
```

备选解释是把“管理层曾作出该表述”作为 speech-act fact_claim。两种语义不能混为一条 Claim，需要大黄裁断。

### D-003：Case A 与 Case C 的 independence_group 不一致

当前：

```text
Case A = smics_official
Case C = smics_annual_report
```

但 Case A 材料自身标注来源为 annual_report，且既有设计已明确 A 是 C 的派生摘要。

建议统一为：

```text
smics_annual_report_2025
```

否则 A/C 会被错误统计为两组独立证据。

### D-004：Case B 出现原文未提及的“中芯国际”实体

SRT 原文仅出现：

```text
联电
格芯
台积电
UP主
```

没有出现中芯国际。

建议：

```text
删除 expected_entity_b_001（中芯国际）
```

除非 Document 元数据中存在明确的中芯国际主题字段，并且 M2-003 设计明确允许从该元数据提取实体。

### D-005：Case B 使用同一观点支持自身，并错误允许 Fact 晋级

当前：

```text
Claim：三家公司保持投入
Evidence：同一句 UP 主陈述
FactCandidate：promotable=true
```

这是循环支持，且只有单一二手观点来源。

建议：

```text
expected_evidence：删除，或改为 direct_quote 且仅证明“UP主作出该陈述”
expected_fc_b_001.promotable=false
rejection_reason=缺少独立可核验来源
```

“可公开查证”不是当前材料中已经存在的 Evidence。

### D-006：Case C Claim 包含原文未支持的 16.5% 增长率

当前 statement：

```text
2025年营收达到673.23亿元，同比增长16.5%
```

当前 source_quote 只支持：

```text
Revenue reached CNY 67.323 billion in 2025.
```

并且 note 已承认 16.5% 是人工计算。

建议：

```text
Claim statement只保留绝对值
```

或另建：

```text
Calculated DataPoint
calculation_method=calculated
calculation_expression=(673.23-577.96)/577.96
input_data_point_ids=[...]
```

在 M2-003A 当前 Schema 未支持计算链前，不应将 16.5% 放入预期 Claim。

### D-007：Case C Prediction 缺少 time_horizon

核心 Claim 模型要求 prediction 必须有 `time_horizon`，但 Expected Results Schema 没有该字段。

建议新增：

```json
"time_horizon": {
  "start": null,
  "end": null,
  "precision": "unknown"
}
```

Schema 应使用条件约束：

```text
if claim_type=prediction
then time_horizon required and non-null
```

## 3. Schema问题

### D-008：Expected Results Schema 过于宽松

当前缺少：

```text
additionalProperties=false
核心枚举enum
SHA-256 pattern
ID最小长度
claim_type条件约束
prediction time_horizon约束
FactCandidate目标互斥约束
promotable/rejection_reason条件约束
reviewed_by/reviewed_at联动
```

还缺少跨引用校验：

```text
entity_id
claimant_id
target_object_id
supporting_evidence_ids
warning target_id
```

JSON Schema不能完成所有跨引用检查，但结构级约束应尽可能严格，其余交给Checker。

### D-009：Schema命名空间不应继续放在 ingestion/v1_1

Expected Results 属于：

```text
M2-003 extraction / QA contract
```

不是 M2-001/002 ingestion contract。

建议路径：

```text
schemas/extraction/v1/expected_results.schema.json
```

或：

```text
schemas/qa/m2_003/v1/expected_results.schema.json
```

保留旧路径可做兼容副本，但顶层 registry 应明确归属。

## 4. Gate 0 Checker问题

### D-010：当前Checker是“字段存在检查”，不是Gate 0语义验证

当前Checker：

- 不调用 JSON Schema；
- 不校验 source hash；
- 不打开原始材料或 ContentUnit snapshot；
- 不验证 quote 是否真的存在；
- 不验证枚举；
- 不验证引用完整性；
- 不检查重复ID；
- 不检查三个固定Case是否齐全；
- 不检查 reviewer字段；
- 不检查 Case A/C independence_group；
- G0-7 只要求至少两类决策，不是三类；
- G0-3 只检查字段存在，不判断候选是否合理。

因此 `21/21` 应改称：

```text
21/21 preliminary structural checks
```

不得称为 Gate 0 PASS。

## 5. 复核状态

三份文件当前均为：

```text
reviewed_by=null
reviewed_at=null
disagreements=[]
```

独立复核完成后应：

```text
reviewed_by=大G
reviewed_at=<timestamp>
disagreements=[D-001 ...]
```

争议裁断后：

```text
resolution
adjudication_note
```

必须填写。

## 6. 建议修订顺序

1. 修订 Case A/B/C Expected Results；
2. 修订 Expected Results Schema；
3. 修订 Gate0 Checker；
4. 运行 JSON Schema + Checker；
5. 婉儿重新QA；
6. 大黄裁断 D-002；
7. 生成正式 Gate 0 报告；
8. Gate 0 PASS 后启动 M2-003B。
