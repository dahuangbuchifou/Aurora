# Aurora M1-002 交付说明

## 一、交付目标

使用中芯国际网页 PR、视频转写和年报 PDF 摘录三类测试素材，验证 Aurora 对象的：

```text
创建
→ Pydantic 校验
→ JSON Schema 校验
→ SQLite 存储
→ 类型恢复
→ 关联查询
→ 来源回溯
```

## 二、重要声明

案例数据来自仓库 `m1_002_materials/` 中的人工测试文本。本任务未对财务数据、价格、估值和观点进行独立外部核验，不构成投资、交易或仓位建议。

## 三、交付内容

- 三个案例的完整结构化 JSON；
- 网页段落、视频时间戳和 PDF 页码定位；
- Case A PR 到 Case C 年报的 Provenance 链；
- 相同财务数据的 `independence_group` 去重标识；
- Claim、Evidence、Fact、Knowledge、Insight、Opinion 和 Output 全链；
- Traceability Engine；
- 悬空引用、重复 ID、依赖循环和来源回溯检查；
- Pydantic 与 JSON Schema 双重校验测试；
- Repository 导入、查询和反序列化测试；
- `test_claim_not_promoted_to_fact` 观点污染测试；
- PR 与年报不一致数据的 `disputed + refute Evidence` 测试；
- 对象模型真实案例审计报告。

## 四、覆盖对象

案例包覆盖 M1-001 的全部 17 类对象：

```text
Source / Document / ContentUnit / Entity / Event / DataPoint
Claim / Evidence / Fact / KnowledgeObject / Relation / TimelineEntry
Insight / PersonalOpinion / OutputArtifact / Feedback / ProcessingRun
```

## 五、核心设计结论

### 1. PR 与年报不算两份独立证据

Case A 与 Case C 中相同财务数据使用相同 `independence_group`。PR 数据通过 `Provenance.origin_object_ids` 指向年报数据点。

### 2. 增长与估值谨慎不是直接冲突

管理层增长目标与 UP 主短期估值谨慎使用 `EvidenceRole.QUALIFY` 表达限定关系，而不是简单标记为 `REFUTE`。

### 3. 明确数据冲突保留而不覆盖

Case A 中“8 英寸利用率低于 70%”与 Case C 测试摘录中的 76.5% 不一致。处理方式为：

```text
Claim.epistemic_status = disputed
Evidence.evidence_role = refute
```

原始内容和两条来源链均保留。

### 4. PersonalOpinion 保持草案

```text
opinion_status = draft
confirmed_by_user = false
```

系统没有替用户确认观点，也没有输出买卖或仓位结论。

## 九、QA 重点

1. 三种来源定位是否准确；
2. 全部对象是否通过 Pydantic 和 JSON Schema；
3. 是否存在悬空引用或重复 ID；
4. PR 与年报是否共享证据独立组；
5. Output 是否能追溯至三个 Source；
6. 视频观点是否没有被升级为 Fact；
7. `disputed + refute` 冲突表达是否合理；
8. PersonalOpinion 是否保持 Draft；
9. DataPoint 的 MAJOR 问题是否完整进入 M1-003 输入。

## 十、不包含

本任务不包含自动抓取、PDF 解析、ASR、LLM 提取、Web API、向量检索、自动投资决策或 WQRS 仓位调整。
