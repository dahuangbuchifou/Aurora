# Aurora 核心理念与认知闭环设计 V1.0

> **项目名称**：Aurora Intelligence Platform  
> **文档类型**：核心理念与认知模型  
> **版本**：V1.0  
> **状态**：正式立项基线  
> **适用阶段**：个人化优先，平台化延伸  
> **核心使命**：通过持续吸收多源信息，将其转化为结构化知识、可检验观点与可复用输出，帮助个人形成并迭代自己的认知体系。

---

## 1. 本章综述

Aurora 不是一个视频转文字工具，也不是传统意义上的文档管理系统、知识库或聊天机器人。

它要解决的核心问题是：

> 在信息来源越来越多、观点彼此冲突、个人注意力有限的环境中，如何持续吸收外部信息，形成自己的结构化知识与独立判断，并将这些判断用于研究、表达和决策。

因此，Aurora 的工作不能停留在“采集—摘要—保存”。

完整链路应当是：

```text
多源信息
    ↓
内容理解
    ↓
事实与观点分离
    ↓
证据核验与冲突比较
    ↓
结构化知识沉淀
    ↓
洞察形成
    ↓
个人观点形成
    ↓
输出与应用
    ↓
结果反馈
    ↓
观点修正与认知升级
```

这条链路称为：

# Aurora 个人认知闭环

Aurora 的最终价值，不取决于系统收集了多少文件，而取决于它是否能够帮助用户：

1. 看见更多有效信息；
2. 分辨事实、观点与推测；
3. 识别不同来源之间的共识和冲突；
4. 形成有依据、可追溯、可修正的个人观点；
5. 将观点转化为报告、研究结论、决策输入或长期知识；
6. 根据后续结果不断修正原有认知。

---

## 2. Aurora 的核心定位

### 2.1 正式定位

Aurora 是一套：

> **个人认知增强与知识操作系统**  
> Personal Cognitive Augmentation & Knowledge Operating System

它以个人为首要服务对象，通过 AI 完成信息采集、内容理解、知识结构化、观点管理、输出生成与反馈更新。

第一阶段，Aurora 优先服务一个真实用户和一组真实任务，而不是一开始建设面向所有人的通用平台。

演进路径为：

```text
个人真实需求
    ↓
个人工作流
    ↓
稳定通用能力
    ↓
模块化服务
    ↓
平台化扩展
```

### 2.2 Aurora 不是什么

Aurora 不是：

- 单纯的视频下载或字幕识别工具；
- 只保存文件的网盘；
- 只支持相似度检索的向量数据库；
- 自动替用户下结论的“万能专家”；
- 不保留证据链的内容总结器；
- 一次性问答后不再积累的聊天机器人；
- 直接替代专业判断和责任承担的决策系统。

### 2.3 Aurora 是什么

Aurora 是：

- 多来源信息的统一入口；
- 事实、观点、证据和推测的分离器；
- 个人结构化知识的长期存储系统；
- 不同观点之间的比较与冲突分析系统；
- 个人观点的形成、版本管理与复核系统；
- 面向 WQRS、AI 研究、学习和写作等应用的知识底座；
- 一个会随着使用时间增长而提升价值的长期认知资产。

## 3. 项目的根本目标

Aurora 的根本目标不是“替用户知道更多”，而是帮助用户形成更高质量的认知过程。

### 3.1 降低信息处理成本

将用户从以下重复劳动中释放出来：

- 一个个下载视频；
- 手工提取字幕；
- 复制网页正文；
- 整理 PDF 和研报；
- 重复阅读相似内容；
- 手工记录人物、时间和观点；
- 在多个收藏夹和文件夹中查找旧资料。

系统负责处理机械工作，人保留判断权。

### 3.2 建立个人结构化知识

外部内容进入 Aurora 后，不应只保存为原文和摘要，而应逐步转化为：

- 人物；
- 机构；
- 公司；
- 行业；
- 技术；
- 事件；
- 时间；
- 事实；
- 数据；
- 观点；
- 论据；
- 反方意见；
- 风险；
- 假设；
- 预测；
- 结论；
- 关联对象。

知识必须可搜索、可组合、可追溯、可更新。

### 3.3 形成个人观点

Aurora 不应把“某位专家的观点”直接当作用户的观点。

系统必须明确区分：

```text
外部来源说了什么
AI 从材料中提取了什么
多个来源形成了什么共识
仍然存在哪些分歧
用户当前认可什么
用户为什么认可
哪些条件会推翻该观点
```

用户观点必须是建立在证据、比较和个人判断基础上的独立对象。

### 3.4 支持持续输出

知识只有进入使用场景，才会产生价值。

Aurora 应支持输出：

- 主题研究摘要；
- 行业研究报告；
- 人物观点追踪；
- 公司背景材料；
- 正反观点矩阵；
- 风险清单；
- 研究备忘录；
- 学习笔记；
- 文章和演讲提纲；
- WQRS 可消费的结构化研究输入；
- 对原有观点的更新提醒。

### 3.5 建立认知复利

随着资料积累，Aurora 不应只是“文件越来越多”，而应表现为：

- 旧知识可以被新任务调用；
- 新信息可以自动关联旧观点；
- 重复内容被识别和合并；
- 观点变化有时间线；
- 预测结果可以被复盘；
- 用户的判断框架逐渐明确；
- 相同问题的研究效率持续提高。

## 4. 核心设计理念

### 4.1 多源，而非单源

任何重要问题都不应只依赖一种渠道或一种声音。

Aurora 支持的信息来源包括但不限于：

- 短视频和长视频；
- 播客与录音；
- 新闻；
- 公司公告；
- 政策文件；
- 券商研报；
- 学术论文；
- 技术博客；
- GitHub 项目；
- 公众号文章；
- 社交媒体观点；
- 网页；
- PDF、Word、Excel 和 Markdown；
- 用户自己的笔记和判断；
- 数据接口与数据库。

多源的目的不是增加数量，而是形成：

```text
交叉验证
观点对照
证据补充
偏差识别
时间演化
```

### 4.2 事实与观点必须分离

Aurora 必须至少区分四类内容：

| 类型 | 定义 | 示例 |
|---|---|---|
| 事实 | 可由外部证据验证的陈述 | 某公司发布了年度报告 |
| 数据 | 具有数值、口径和时间的信息 | 营收同比增长 18% |
| 观点 | 某个主体的解释或判断 | 该增长具有可持续性 |
| 预测 | 对未来状态的判断 | 未来两年需求将继续增长 |

系统不得将观点包装成事实，也不得把未经验证的预测当作确定结论。

### 4.3 来源必须可追溯

任何结构化知识、洞察和观点都应尽可能保留：

- 原始来源；
- 原始链接或文件；
- 作者或发布机构；
- 发布时间；
- 采集时间；
- 原文位置或时间戳；
- 内容版本；
- 提取方式；
- 相关证据；
- 引用关系。

无法追溯来源的内容，可以作为线索，但不能作为高置信度结论。

### 4.4 AI 负责增强，不负责夺权

Aurora 中 AI 的职责是：

- 提高采集和整理效率；
- 提取结构；
- 发现关联；
- 识别冲突；
- 提醒遗漏；
- 辅助形成观点；
- 生成可审阅输出。

最终的个人观点、重要结论和高风险决策，仍由用户确认。

系统必须允许用户：

- 接受；
- 修改；
- 否决；
- 标记存疑；
- 添加个人备注；
- 提升或降低置信度；
- 指定复核条件。

### 4.5 观点必须可修正

Aurora 不追求永远正确，而追求：

> 错误能够被发现，观点能够被修正，修正过程能够被记录。

每个重要观点应包含：

- 当前结论；
- 支持证据；
- 反对证据；
- 关键假设；
- 当前置信度；
- 有效时间范围；
- 失效条件；
- 下次复核时间；
- 历史版本。

### 4.6 输出必须服务真实用途

系统不为“看起来聪明”而生成内容。

所有输出应明确用途，例如：

- 用于快速了解；
- 用于深入研究；
- 用于 WQRS 输入；
- 用于撰写文章；
- 用于学习复习；
- 用于会议准备；
- 用于判断是否继续投入研究；
- 用于跟踪某个观点是否发生变化。

不同用途应调用不同模板和证据标准。

## 5. Aurora 认知闭环模型

Aurora 的核心不是某个单一功能，而是以下闭环。

## 5.1 第一阶段：信息输入（Information Intake）

用户通过主动提交或自动订阅，将外部信息引入系统。

输入可以是：

```text
链接
文件
音视频
订阅源
API 数据
用户笔记
研究问题
```

本阶段目标：

- 保留原始内容；
- 记录来源和时间；
- 判断内容类型；
- 完成基础去重；
- 进入待处理队列。

输出对象：

```text
Source Object
Raw Document
```

## 5.2 第二阶段：内容理解（Understanding）

系统对原始内容进行解析和理解。

典型处理包括：

- 音视频转写；
- 网页正文提取；
- PDF 和 Office 内容解析；
- 内容分段；
- 标题与章节识别；
- 人物和机构识别；
- 主题识别；
- 摘要；
- 关键句提取；
- 事实、数据、观点和预测分类；
- 论据和风险提取。

输出对象：

```text
Parsed Document
Content Units
```

## 5.3 第三阶段：证据整理（Evidence Structuring）

系统把内容拆成可验证单元，并建立证据关系。

每条重要陈述至少记录：

- 谁提出；
- 何时提出；
- 基于什么依据；
- 来源是否一手；
- 是否存在其他来源支持；
- 是否存在反例；
- 是否存在利益冲突；
- 是否已经过时。

输出对象：

```text
Fact Object
Data Point
Claim Object
Evidence Object
```

## 5.4 第四阶段：知识沉淀（Knowledge Formation）

系统将零散信息组织为可复用知识。

知识不等同于摘要。

知识应包含：

- 清晰主题；
- 结构化对象；
- 实体关系；
- 时间关系；
- 来源关系；
- 版本信息；
- 适用范围；
- 关联概念；
- 当前状态。

输出对象：

```text
Knowledge Object
Entity
Event
Relation
Timeline
```

## 5.5 第五阶段：洞察形成（Insight Generation）

洞察是系统在多条知识之间建立联系后形成的高阶认识。

典型洞察包括：

- 多个来源形成的共同趋势；
- 表面无关事件之间的潜在关系；
- 某个观点支持度正在上升或下降；
- 产业瓶颈正在从一个环节转移到另一个环节；
- 某项数据与主流叙事不一致；
- 某个风险被大量来源忽略；
- 某个假设已被新事实削弱。

洞察必须明确：

- 基于哪些知识；
- 推理链是什么；
- 哪些部分是事实；
- 哪些部分是推断；
- 当前置信度；
- 可能的替代解释。

输出对象：

```text
Insight Object
```

## 5.6 第六阶段：个人观点形成（Opinion Formation）

个人观点是用户基于事实、知识、洞察和自身目标形成的阶段性判断。

个人观点不是 AI 自动生成后直接生效，而应经历：

```text
AI 提出观点草案
    ↓
列出支持与反对证据
    ↓
说明关键假设
    ↓
用户审阅与修改
    ↓
用户确认
    ↓
进入个人观点库
```

每个个人观点应包含：

```text
观点标题
当前结论
适用领域
形成日期
支持证据
反对证据
关键假设
未知变量
置信度
失效条件
复核周期
用户备注
历史版本
```

输出对象：

```text
Personal Opinion Object
```

## 5.7 第七阶段：输出与应用（Output & Application）

个人知识和观点可被不同应用调用。

典型输出：

- 一页式主题简报；
- 完整研究报告；
- 观点矩阵；
- 时间线；
- 风险清单；
- 公司或行业档案；
- 决策备忘录；
- 学习卡片；
- 问答结果；
- WQRS 结构化输入；
- 写作素材包。

输出必须标明：

- 内容用途；
- 数据截止时间；
- 主要来源；
- 观点与事实的边界；
- 不确定性；
- 风险提示。

## 5.8 第八阶段：结果反馈（Feedback）

系统记录观点和输出在现实中的后续表现。

反馈来源包括：

- 新事实出现；
- 预测到期；
- 用户实际决策结果；
- 研究结论被证伪；
- 某个关键假设成立或失效；
- 用户主动修改认知；
- 应用系统返回结果。

输出对象：

```text
Feedback Object
Outcome Object
```

## 5.9 第九阶段：认知修正（Cognitive Update）

根据反馈，Aurora 触发：

- 观点维持；
- 观点降级；
- 观点升级；
- 观点修订；
- 观点失效；
- 重新研究；
- 新增反例；
- 调整来源可信度；
- 更新个人方法论。

至此形成完整闭环：

```text
信息
→ 理解
→ 证据
→ 知识
→ 洞察
→ 观点
→ 输出
→ 反馈
→ 修正
→ 新认知
```

## 6. 五层认知对象模型

Aurora 第一阶段应重点区分以下五层对象。

### 6.1 Information：信息

定义：

未经 Aurora 判断的外部输入。

示例：

- 一段视频；
- 一篇新闻；
- 一份公告；
- 一条社交媒体发言；
- 一份研究报告。

特点：

- 保留原貌；
- 不代表真实；
- 不代表用户认可；
- 是后续处理的原材料。

### 6.2 Knowledge：知识

定义：

经过解析、结构化和来源标注的内容。

示例：

```json
{
  "entity": "某公司",
  "event": "发布新一代产品",
  "event_date": "2026-06-01",
  "source_type": "company_announcement",
  "evidence_level": "primary"
}
```

特点：

- 可搜索；
- 可关联；
- 可验证；
- 有时间和来源。

### 6.3 Insight：洞察

定义：

基于多条知识之间的关系形成的解释或发现。

示例：

> 多个产业链来源显示，当前限制产能扩张的主要因素正在从芯片供应转向电力与散热。

特点：

- 包含推理；
- 不一定是确定事实；
- 必须展示依据；
- 必须标记置信度。

### 6.4 Opinion：个人观点

定义：

用户基于证据、洞察和个人目标确认的阶段性判断。

示例：

> 当前阶段，AI 基础设施仍处于扩张周期，但产业链机会可能由单一算力环节向电力、散热和网络侧扩散。

特点：

- 属于用户；
- 有形成时间；
- 有适用范围；
- 有反方证据；
- 可以被修正。

### 6.5 Output：输出

定义：

针对具体用途，由知识和观点生成的可使用成果。

示例：

- WQRS 行业背景输入；
- 机器人行业研究报告；
- 会议准备材料；
- 文章提纲；
- 学习计划。

特点：

- 面向任务；
- 有受众；
- 有格式；
- 有截止时间；
- 有证据标准。

## 7. 多源观点管理机制

Aurora 的重要能力不是把不同声音平均混合，而是保留差异并帮助用户理解差异。

### 7.1 观点主体识别

每条观点应尽量识别：

- 发言人；
- 所属机构；
- 专业背景；
- 立场；
- 可能的利益关系；
- 观点发布时间；
- 观点适用周期。

### 7.2 观点结构拆解

每条观点至少拆为：

```text
结论
依据
关键假设
预测周期
受影响对象
潜在风险
可验证条件
```

### 7.3 正反观点矩阵

对重要主题，系统应形成：

| 维度 | 支持方 | 反对方 | 当前状态 |
|---|---|---|---|
| 市场空间 | 需求长期增长 | 商业化速度低于预期 | 尚未定论 |
| 技术可行性 | 核心部件持续进步 | 成本和可靠性仍受限 | 部分验证 |
| 投资价值 | 产业趋势明确 | 估值可能提前透支 | 高度依赖价格 |

### 7.4 共识不等于真相

来源数量较多只能说明某观点传播广泛，不代表它必然正确。

系统应区分：

- 多数来源重复同一原始消息；
- 多个独立来源得出相似结论；
- 同一机构不同渠道重复表达；
- 真正独立的证据交叉验证。

### 7.5 少数观点保护

Aurora 不应因为某个观点处于少数就自动过滤。

对于具有明确依据、可检验条件或专业背景支持的少数意见，应保留为：

```text
Contrarian View
Minority Hypothesis
Risk Scenario
```

它们可能成为未来认知修正的重要线索。

---

## 8. 个人观点管理机制

### 8.1 观点状态

个人观点建议设置以下状态：

| 状态 | 含义 |
|---|---|
| Draft | AI 或用户提出的观点草案 |
| Under Review | 正在补充证据与反方意见 |
| Active | 用户当前认可的有效观点 |
| Watch | 有一定依据，但关键条件尚未确认 |
| Challenged | 新证据正在挑战原观点 |
| Revised | 已完成修订 |
| Invalidated | 已被证伪或不再适用 |
| Archived | 仅保留历史价值 |

### 8.2 观点置信度

置信度不是“正确概率”的精确数学值，而是对当前证据强度和不确定性的结构化表达。

可采用五级：

| 等级 | 含义 |
|---|---|
| Very Low | 仅有线索，证据不足 |
| Low | 有部分支持，但反例明显 |
| Medium | 支持与反对证据相对均衡 |
| High | 多个独立来源支持，关键假设较稳固 |
| Very High | 有强一手证据并经长期验证 |

第一阶段不追求伪精确的 73%、86% 等数字，避免让主观判断披上数学外衣。

### 8.3 观点失效条件

每个重要观点都应回答：

> 出现什么情况时，我将不再坚持这个观点？

例如：

```text
若核心产品连续两个季度出货低于预期；
若主要政策支持取消；
若单位成本未按计划下降；
若竞争格局发生根本变化；
若关键数据被证伪。
```

### 8.4 观点复核机制

观点复核可由以下事件触发：

- 到达预定复核日期；
- 出现重大新信息；
- 数据与原假设明显偏离；
- 反方证据显著增加；
- 相关应用发出风险信号；
- 用户主动发起复盘。

## 9. 输出体系设计

Aurora 的输出不应只有一种“总结”。

### 9.1 快速理解输出

适用于用户第一次接触主题。

内容：

- 一句话结论；
- 三个核心事实；
- 主要分歧；
- 关键风险；
- 是否值得继续研究。

### 9.2 深度研究输出

适用于行业、公司、技术或政策研究。

内容：

- 研究范围；
- 背景；
- 核心事实；
- 数据；
- 多方观点；
- 正反论证；
- 关键假设；
- 未知变量；
- 当前洞察；
- 个人观点草案；
- 后续跟踪事项。

### 9.3 观点输出

适用于形成个人表达。

内容：

- 我的结论；
- 为什么；
- 我依赖哪些证据；
- 哪些地方不确定；
- 反方观点是什么；
- 什么会让我改变看法。

### 9.4 应用接口输出

面向 WQRS 等系统时，输出必须结构化。

例如：

```json
{
  "topic": "人形机器人",
  "as_of_date": "2026-06-10",
  "consensus": [],
  "contrarian_views": [],
  "key_facts": [],
  "risks": [],
  "user_opinion": {},
  "source_ids": [],
  "confidence": "Medium"
}
```

### 9.5 输出纪律

任何正式输出必须尽量说明：

- 信息截止日期；
- 来源范围；
- 是否存在资料缺口；
- 事实与推断的边界；
- 当前置信度；
- 不构成专业决策替代。

## 10. 典型应用闭环

### 10.1 主题研究场景

用户提出：

> 研究人形机器人未来三年的产业变化。

Aurora 执行：

```text
确定研究范围
→ 收集视频、研报、论文、公告和新闻
→ 提取事实、数据和观点
→ 识别产业链和关键人物
→ 构建正反观点矩阵
→ 提炼核心变量
→ 形成洞察
→ 生成个人观点草案
→ 用户确认
→ 输出研究报告
→ 持续跟踪新证据
```

### 10.2 人物观点追踪

用户关注某位专家或企业管理者。

系统记录：

- 其历史观点；
- 观点变化；
- 使用过的依据；
- 哪些预测已经实现；
- 哪些预测未实现；
- 与当前观点是否矛盾。

### 10.3 WQRS 接入

Aurora 为 WQRS 提供：

- 行业叙事变化；
- 公司背景事件；
- 管理层观点；
- 政策变化；
- 市场主流观点；
- 反方意见；
- 证据可信度；
- 观点时效性。

WQRS 负责金融专业判断和风险约束，Aurora 不直接替代投资决策。

### 10.4 写作与表达

用户准备输出一篇文章时，Aurora 可以生成：

- 观点大纲；
- 证据包；
- 反方意见；
- 关键引用；
- 数据表；
- 逻辑漏洞提示；
- 不确定性说明。

输出仍体现用户观点，而不是拼接外部内容。

## 11. 用户在系统中的职责

Aurora 并不意味着用户可以完全退出认知过程。

用户需要承担五类核心工作。

### 11.1 定义关注范围

明确：

- 哪些主题值得长期跟踪；
- 哪些来源值得优先采集；
- 哪些信息属于噪音；
- 哪些领域需要严格证据标准。

### 11.2 定义价值标准

告诉系统：

- 什么信息对自己有用；
- 什么叫高质量来源；
- 什么样的观点值得保留；
- 哪些风险必须重点记录；
- 哪些结论必须经过人工确认。

### 11.3 审核重要观点

对于高影响结论，用户必须：

- 审阅依据；
- 查看反方证据；
- 确认关键假设；
- 决定是否纳入个人观点库。

### 11.4 提供反馈

用户需要标记：

- 有用；
- 无用；
- 重复；
- 错误；
- 过时；
- 值得深挖；
- 应加入长期跟踪。

### 11.5 对最终应用负责

Aurora 提供认知辅助，但不替代用户承担：

- 投资责任；
- 法律责任；
- 医疗责任；
- 商业决策责任；
- 公开表达责任。

## 12. 第一阶段范围与非目标

### 12.1 第一阶段必须实现

第一阶段应完成一个最小认知闭环：

```text
多来源输入
→ 自动解析
→ 事实与观点提取
→ 结构化知识卡片
→ 主题检索
→ 正反观点整理
→ 个人观点草案
→ Markdown 输出
```

优先支持：

- 网页；
- PDF；
- Markdown；
- 视频或音频链接；
- 本地音视频；
- 用户手工笔记。

### 12.2 第一阶段暂不追求

- 覆盖所有互联网平台；
- 绕过平台权限和反爬限制；
- 建设复杂通用知识图谱；
- 自动执行高风险决策；
- 自研基础大模型；
- 多人团队协作；
- 完整商业化权限体系；
- 炫目的前端界面；
- 大规模实时数据处理；
- 全自动“无人认知”。

### 12.3 第一阶段核心原则

> 先证明认知闭环有价值，再扩展数据源和平台能力。

## 13. 成功标准

Aurora V1.0 的成功，不以代码量和功能数量衡量。

### 13.1 输入成功

用户可以将不同渠道的内容方便地提交给系统，无需重复手工处理。

### 13.2 理解成功

系统能够正确区分：

- 事实；
- 数据；
- 观点；
- 预测；
- 风险；
- 未知变量。

### 13.3 知识成功

同一主题的不同来源能够被关联，而不是形成孤立文件。

### 13.4 观点成功

系统能够帮助用户生成一份包含支持证据、反方意见、关键假设和失效条件的个人观点草案。

### 13.5 输出成功

用户能够基于 Aurora 的知识产出：

- 研究报告；
- 文章；
- 决策备忘录；
- WQRS 输入；
- 学习总结。

### 13.6 复利成功

系统使用一段时间后，应体现：

- 研究相同主题所需时间下降；
- 历史知识调用率上升；
- 重复信息处理减少；
- 用户观点变化可追溯；
- 结论质量和表达质量提升。

## 14. 持续升级路径

Aurora 的升级应围绕能力，而不是盲目堆功能。

### Level 1：信息处理

能力：

- 采集；
- 转写；
- 解析；
- 摘要；
- 标签。

目标：

让信息进入系统。

### Level 2：知识结构化

能力：

- 实体；
- 事件；
- 数据；
- 观点；
- 来源；
- 时间线；
- 混合检索。

目标：

让系统记得并找得到。

### Level 3：洞察与观点

能力：

- 多源比较；
- 正反观点；
- 证据强度；
- 冲突识别；
- 个人观点管理；
- 观点版本。

目标：

让系统帮助用户形成判断。

### Level 4：主动跟踪

能力：

- 主题监控；
- 观点变化提醒；
- 假设失效提醒；
- 定期复核；
- 自动简报。

目标：

让系统主动辅助。

### Level 5：应用与平台

能力：

- WQRS 接入；
- AI 研究助手；
- 学习助手；
- 写作助手；
- API；
- SDK；
- Agent。

目标：

从个人系统抽象为可复用平台。

---

## 15. 核心纪律

Aurora 必须长期坚持以下纪律：

1. **不把摘要当知识。**
2. **不把观点当事实。**
3. **不把共识当真相。**
4. **不把 AI 生成内容当证据。**
5. **不隐藏来源和不确定性。**
6. **不因信息数量多而提高置信度。**
7. **不让旧观点在无复核情况下永久有效。**
8. **不为追求自动化而取消人的最终判断。**
9. **不优先建设低频、炫技但无实际价值的功能。**
10. **每一项能力都必须服务真实认知和输出。**

---

## 16. 本章结论

Aurora 的核心使命可以概括为：

> 从不同渠道收集不同的声音、信息和观点，经过理解、验证、关联与沉淀，帮助用户形成自己的结构化知识和个人观点，并将其持续输出、应用和修正。

它的核心闭环是：

```text
世界信息
    ↓
多源采集
    ↓
AI 理解
    ↓
事实与观点分离
    ↓
证据与冲突分析
    ↓
结构化知识
    ↓
洞察
    ↓
个人观点
    ↓
输出与应用
    ↓
现实反馈
    ↓
观点修正
    ↓
认知升级
```

Aurora 最终不是替用户思考。

它要做的是：

> 让用户的思考有更好的输入、更清晰的结构、更完整的证据、更可靠的记忆，以及持续自我修正的能力。

这就是 Aurora 与普通知识库、普通 RAG 系统和一次性 AI 问答工具之间最根本的区别。
