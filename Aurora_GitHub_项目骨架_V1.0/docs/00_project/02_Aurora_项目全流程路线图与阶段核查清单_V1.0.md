# Aurora 项目全流程路线图与阶段核查清单 V1.0

> **项目名称**：Aurora Intelligence Platform  
> **定位**：个人认知增强与知识操作系统  
> **当前策略**：先个人化，再平台化；先完成最小认知闭环，再扩展能力  
> **适用对象**：个人开发者 + AI 辅助开发  
> **建议技术基础**：Python、FastAPI、SQLite/PostgreSQL、对象存储、本地或云端 LLM、Whisper、向量检索  
> **路线原则**：每个阶段必须“可验证、可回退、可复用”，未通过阶段验收，不进入下一阶段

---

# 一、路线图总览

Aurora 的建设不应从“搭平台”开始，而应按以下顺序推进：

```text
01 项目边界与成功标准
        ↓
02 用户场景与需求优先级
        ↓
03 核心对象模型与知识标准
        ↓
04 MVP最小认知闭环
        ↓
05 多源采集能力
        ↓
06 内容解析能力
        ↓
07 AI理解与证据抽取
        ↓
08 知识存储与检索
        ↓
09 洞察与个人观点管理
        ↓
10 输出、工作流与主动跟踪
        ↓
11 WQRS等应用接入
        ↓
12 稳定性、安全与运维
        ↓
13 模块化与平台化
```

整个项目建议分为三大阶段：

| 大阶段 | 目标 | 典型周期 |
|---|---|---:|
| A. 概念与标准阶段 | 把“要做什么”定义清楚 | 2～4周 |
| B. MVP闭环阶段 | 跑通一条真实认知链路 | 6～12周 |
| C. 稳定与扩展阶段 | 多来源、自动化、应用接入 | 3～12个月 |

> 以上周期以“1人主导、AI辅助、每周投入约10～15小时”为假设。若全职开发，周期可缩短；若同时承担WQRS等项目，周期应适当延长。

# 二、Phase 0：项目治理与范围冻结

## 目标

建立 Aurora 的“项目宪法”，防止后续功能膨胀、方向漂移和重复返工。

## 主要任务

1. 固化项目使命、边界和非目标。
2. 明确第一阶段只服务个人使用。
3. 确定第一批真实场景：
   - 多来源资料进入；
   - 事实/观点/预测提取；
   - 主题知识卡片；
   - 正反观点矩阵；
   - 个人观点草案；
   - Markdown输出。
4. 定义阶段验收规则。
5. 建立文档版本、需求变更和缺陷记录规范。

## 任务量

- 文档整理：2～3天
- 需求讨论：2～4次
- 项目目录与版本规范：1天
- 总量：约3～5人日

## 主要难点

- 容易把长期愿景误当成MVP范围；
- 容易将“想做的功能”误当成“必须做的能力”；
- 容易在技术选型前没有定义验收标准。

## 核查项目

- [ ] 是否明确 Aurora 不直接替代专业决策；
- [ ] 是否明确个人化优先；
- [ ] 是否列出第一阶段不做事项；
- [ ] 是否明确每个阶段的进入条件与退出条件；
- [ ] 是否建立统一项目术语表；
- [ ] 是否建立变更记录文件。

## 阶段产物

```text
00_项目定义与路线图.md
01_核心理念与认知闭环设计.md
02_术语表与系统边界.md
CHANGELOG.md
DECISIONS.md
```

## 退出条件

任何参与者都能用一段话回答：

> Aurora解决什么问题、第一阶段做什么、什么暂时不做、如何判断成功。

# 三、Phase 1：用户场景与需求优先级

## 目标

把抽象愿景转化为具体、高频、可验证的使用场景。

## 主要任务

1. 梳理个人真实使用场景。
2. 按“频率 × 价值 × 可实现性”排序。
3. 定义每个场景的输入、处理、输出和失败方式。
4. 确定首个MVP场景。

## 建议首个MVP场景

### 多来源主题研究助手

输入：

- 视频链接或音频；
- 网页文章；
- PDF；
- 用户笔记。

输出：

- 内容摘要；
- 事实；
- 数据；
- 观点；
- 预测；
- 风险；
- 正反观点；
- 个人观点草案；
- Markdown研究简报。

## 任务量

- 场景梳理：2～3天
- 用户旅程设计：2天
- 优先级评审：1天
- 验收用例编写：2～3天
- 总量：约5～8人日

## 主要难点

- 场景描述容易过宽；
- 输出容易停留在“摘要”；
- 很难提前定义什么是“好观点”；
- 不同主题对证据要求不同。

## 核查项目

- [ ] 每个场景是否有明确输入；
- [ ] 是否有明确输出格式；
- [ ] 是否有失败和降级策略；
- [ ] 是否能用真实资料测试；
- [ ] 是否能在10分钟内说明用户价值；
- [ ] 是否选定唯一P0场景。

## 阶段产物

```text
03_用户场景与需求分析.md
04_MVP用户旅程.md
05_需求优先级矩阵.md
tests/use_cases/
```

## 退出条件

选定一个可在8～12周内跑通的完整场景，不再同时开发多个应用。

# 四、Phase 2：核心对象模型与知识标准

## 目标

定义 Aurora 内部“信息如何存在”，这是后续所有采集、理解、检索和输出的基础。

## 核心对象

```text
Source
Document
ContentUnit
Fact
DataPoint
Claim
Evidence
Entity
Event
Relation
KnowledgeObject
Insight
PersonalOpinion
OutputArtifact
Feedback
```

## 主要任务

1. 定义对象字段、ID和关系。
2. 定义事实、观点、预测和推断的边界。
3. 定义来源等级和证据等级。
4. 定义时间、版本、置信度和失效条件。
5. 定义 JSON Schema。
6. 定义最小可用字段和可选字段。
7. 定义去重键与内容指纹。

## 任务量

- 对象建模：5～8天
- JSON Schema：3～5天
- 样例数据：2～3天
- 评审与修订：2～4天
- 总量：约10～15人日

## 主要难点

- 模型太简单，后续无法表达观点关系；
- 模型太复杂，MVP无法落地；
- “置信度”容易伪精确；
- 事实、推断和用户观点容易混淆；
- 同一内容在不同场景下可能有不同含义。

## 核查项目

- [ ] 是否能表达原始来源；
- [ ] 是否能定位原文位置或音视频时间戳；
- [ ] 是否能区分事实、观点、预测和推断；
- [ ] 是否能表达支持证据和反对证据；
- [ ] 是否能保存个人观点版本；
- [ ] 是否支持失效条件和复核日期；
- [ ] 是否有至少20条真实样例通过建模；
- [ ] Schema是否可由Python直接校验。

## 阶段产物

```text
06_核心对象模型.md
schemas/*.json
examples/*.json
07_来源与证据分级规范.md
08_标签与主题体系.md
```

## 退出条件

至少使用20条来自不同渠道的真实内容完成结构化，不出现无法表达的核心信息。

# 五、Phase 3：MVP技术底座

## 目标

建立最小但可靠的工程骨架，避免后期重构成本过高。

## 建议架构

```text
CLI / 简易Web
      ↓
Task Queue
      ↓
Collector
      ↓
Parser
      ↓
Understanding
      ↓
Knowledge Store
      ↓
Search / Export
```

## 主要任务

1. 初始化代码仓库和目录。
2. 建立配置管理。
3. 建立日志、异常和任务状态。
4. 建立数据库和迁移机制。
5. 建立文件存储规范。
6. 建立最小API或CLI。
7. 建立测试框架。
8. 建立开发与生产环境区分。

## 建议第一阶段技术

- Python 3.11
- FastAPI
- SQLite（MVP）/ PostgreSQL（扩展）
- Pydantic
- SQLAlchemy
- Alembic
- FFmpeg
- Whisper/Faster-Whisper
- 本地文件存储
- 向量库后置，不急于第一天引入
- pytest
- Docker可后置

## 任务量

- 工程骨架：4～6天
- 数据库与迁移：3～5天
- 日志与任务状态：2～3天
- 测试基础：2～3天
- 总量：约10～15人日

## 主要难点

- 一开始过度微服务化；
- 数据库结构随对象模型频繁变化；
- 任务失败后无法恢复；
- 原始文件、解析文本和结构化结果混放。

## 核查项目

- [ ] 单个任务是否有唯一ID；
- [ ] 每一步是否记录状态；
- [ ] 失败是否可以重试；
- [ ] 原始资料是否不可覆盖；
- [ ] 配置和密钥是否与代码分离；
- [ ] 数据库迁移是否可回滚；
- [ ] 是否有基础单元测试；
- [ ] 是否能在新电脑上按文档启动。

## 阶段产物

```text
aurora/
config/
data/raw/
data/parsed/
data/structured/
tests/
README.md
.env.example
```

## 退出条件

能够通过一条命令提交一份本地文本或Markdown，并在数据库中生成完整任务记录。

# 六、Phase 4：采集引擎 Collector

## 目标

建立多来源内容的统一入口，但不追求一次支持所有平台。

## 第一批建议支持

### P0

- 本地TXT/Markdown；
- PDF；
- 普通网页URL；
- 本地音视频；
- 可直接访问的视频/音频URL；
- 用户手工笔记。

### P1

- RSS；
- YouTube/B站等可稳定获取的来源；
- GitHub；
- 公众号导出内容；
- 新闻API。

### P2

- 抖音、视频号、小红书等受限平台；
- 登录态采集；
- 浏览器插件；
- 自动订阅监控。

## 主要任务

1. 统一 Collector 接口。
2. URL规范化。
3. 下载和缓存。
4. 内容指纹与去重。
5. 失败重试。
6. 版权和访问权限提示。
7. 来源元数据提取。
8. 插件化适配器。

## 任务量

- 统一接口：3～4天
- 本地文件：2天
- 网页：3～5天
- PDF入口：2～3天
- 音视频入口：4～7天
- 去重与重试：3～4天
- 总量：约15～25人日

## 主要难点

- 平台反爬和链接时效；
- 登录和Cookie管理；
- 视频平台规则变化；
- 文件重复；
- 网页正文和广告混杂；
- 版权与使用边界。

## 核查项目

- [ ] 每个来源是否保留原始URL；
- [ ] 是否记录采集时间；
- [ ] 是否保存MIME类型和文件哈希；
- [ ] 重复提交是否被识别；
- [ ] 采集失败是否给出明确原因；
- [ ] 是否有超时与重试上限；
- [ ] 是否尊重公开访问和版权边界；
- [ ] 不可采集时是否允许用户手工上传。

## 阶段产物

```text
collector/base.py
collector/file.py
collector/web.py
collector/media.py
collector/pdf.py
collector/registry.py
```

## 退出条件

至少支持3类来源稳定进入系统，连续处理50个真实输入，成功率达到可接受水平且失败可解释。

# 七、Phase 5：内容解析引擎 Parser

## 目标

把不同来源统一转换为可理解、可定位的标准文本单元。

## 主要任务

1. 文本编码和清洗。
2. PDF正文、页码与段落提取。
3. 音视频转写与时间戳。
4. 网页正文、标题、作者和发布时间提取。
5. 分段与章节识别。
6. 表格、列表和引用保留。
7. 低质量解析标记。
8. 解析结果版本管理。

## 任务量

- 文本/Markdown：2天
- 网页：3～5天
- PDF：5～10天
- 音视频ASR：5～8天
- 分段和质量评估：4～6天
- 总量：约18～30人日

## 主要难点

- 扫描版PDF；
- 表格、公式和复杂排版；
- 音频噪声和多人对话；
- 人名、术语识别错误；
- 时间戳与原文位置丢失；
- 中文标点和说话人切分。

## 核查项目

- [ ] 解析结果是否可定位回原文；
- [ ] 音视频是否保留时间戳；
- [ ] PDF是否保留页码；
- [ ] 解析失败是否标记质量；
- [ ] 是否保留原始文本和清洗文本；
- [ ] 是否避免重复转写；
- [ ] 是否对长文进行稳定分段；
- [ ] 是否能人工抽检10%样本。

## 阶段产物

```text
parser/text.py
parser/web.py
parser/pdf.py
parser/asr.py
parser/chunker.py
parser/quality.py
```

## 退出条件

选定的MVP来源中，文本解析准确度足以支持后续事实与观点提取，且每条内容可追溯到原文。

# 八、Phase 6：AI理解与证据抽取引擎

## 目标

从“有文本”升级到“能理解”，这是 Aurora 的核心竞争力。

## 主要任务

1. 摘要生成。
2. 主题和实体提取。
3. 事实、数据、观点、预测分类。
4. 发言主体和观点归属。
5. 论据、反例和风险提取。
6. 时间范围和适用条件提取。
7. 结构化JSON输出。
8. Schema校验和自动修复。
9. 来源引用和证据定位。
10. LLM提示词版本管理。
11. 人工评测集建立。

## 任务量

- 提示词原型：5～8天
- 结构化输出：4～6天
- 证据定位：5～8天
- 评测集：5～10天
- 迭代优化：10～20天
- 总量：约25～45人日

## 主要难点

- LLM幻觉；
- 观点归属错误；
- 将推测误判为事实；
- 长内容遗漏；
- 引用无法定位；
- 不同模型输出不一致；
- 置信度伪精确；
- 多人对话区分。

## 核查项目

- [ ] 每条事实是否有来源；
- [ ] 每条观点是否有明确主体；
- [ ] 预测是否标注时间范围；
- [ ] 推断是否明确标识；
- [ ] JSON是否通过Schema；
- [ ] 是否保留模型和提示词版本；
- [ ] 是否有人工标注评测集；
- [ ] 是否统计准确率、遗漏率和幻觉率；
- [ ] 低置信结果是否进入人工复核；
- [ ] 是否支持重新处理旧内容。

## 阶段产物

```text
understanding/prompts/
understanding/extractors/
understanding/validators/
evaluation/gold_set/
evaluation/reports/
```

## 退出条件

在人工标注样本上，系统能够稳定区分事实、观点、预测和风险；重大错误有明确拦截机制。

# 九、Phase 7：知识存储、搜索与关联

## 目标

使知识能够长期保存、快速检索、跨来源关联，而不是形成新的文件坟场。

## 主要任务

1. 结构化数据库设计。
2. 全文搜索。
3. 语义搜索。
4. 标签和实体过滤。
5. 时间范围检索。
6. 来源追溯。
7. 去重与相似观点合并。
8. 主题聚类。
9. 关系与时间线。
10. 检索结果重排。

## 建议顺序

```text
SQL结构化检索
    ↓
全文检索
    ↓
向量检索
    ↓
混合检索
    ↓
关系图谱
```

不要一开始直接上复杂知识图谱。

## 任务量

- SQL存储：5～8天
- 全文搜索：3～5天
- 向量检索：5～8天
- 混合检索：5～8天
- 关联与时间线：7～12天
- 总量：约25～40人日

## 主要难点

- 分块导致上下文丢失；
- 向量相似不等于事实相关；
- 同义词、别名和实体消歧；
- 历史观点与最新观点混淆；
- 过期知识仍被召回；
- 来源重复导致“伪共识”。

## 核查项目

- [ ] 查询是否可按时间过滤；
- [ ] 是否优先最新有效版本；
- [ ] 是否能返回原始来源；
- [ ] 是否能区分独立来源和重复转载；
- [ ] 是否支持人物、公司、主题和事件搜索；
- [ ] 是否能显示观点演变；
- [ ] 是否有检索评测集；
- [ ] 是否记录无结果和错误召回案例。

## 阶段产物

```text
knowledge/repository.py
search/fulltext.py
search/vector.py
search/hybrid.py
search/rerank.py
knowledge/timeline.py
```

## 退出条件

用户能用自然语言查询一个主题，系统返回相关事实、观点、来源和时间，并能追溯到原文。

---

# 十、Phase 8：洞察与个人观点引擎

## 目标

从“知识库”升级为“认知系统”。

## 主要任务

1. 多来源观点聚合。
2. 正反观点矩阵。
3. 共识与独立证据识别。
4. 少数观点保护。
5. 关键假设提取。
6. 未知变量识别。
7. 洞察草案生成。
8. 个人观点草案生成。
9. 用户确认和修改。
10. 观点版本、状态和失效条件。
11. 定期复核和证据更新。

## 任务量

- 观点聚合：5～8天
- 正反矩阵：4～6天
- 洞察生成：7～12天
- 个人观点工作流：7～12天
- 版本与复核：5～8天
- 总量：约25～40人日

## 主要难点

- 把多数意见误当真相；
- AI替用户“偷做决定”；
- 观点缺乏反方证据；
- 观点长期不复核；
- 新旧观点冲突；
- 无法解释洞察推理链。

- [ ] 个人观点是否需用户确认；
- [ ] 是否有失效条件；
- [ ] 是否有复核日期；
- [ ] 是否保留历史版本；
- [ ] 是否能显示观点变化原因；
- [ ] 是否禁止AI生成内容充当外部证据。

## 阶段产物

```text

insight/contradiction.py
opinion/workflow.py
opinion/versioning.py
opinion/review.py
```

## 退出条件

对一个真实主题，系统能产出结构完整的个人观点草案，并让用户明确看到支持、反对、假设和不确定性。

# 十二、Phase 10：WQRS 接入验证

## 目标

验证 Aurora 能否作为独立底座，为专业应用提供可靠知识输入。

## 建议首个接口

Aurora 输出：

```json
{
  "topic": "某行业或公司",
  "as_of_date": "YYYY-MM-DD",
  "key_facts": [],
  "events": [],
  "consensus_views": [],
  "contrarian_views": [],
  "risks": [],
  "user_opinion": {},
  "source_ids": [],
  "confidence": "Medium"
}
```

WQRS 消费：

- 行业背景；
- 政策变化；
- 公司事件；
- 管理层观点；
- 市场叙事变化；
- 风险线索；
- 信息时效性。

## 主要任务

1. 定义接口契约。
2. 定义WQRS可消费字段。
3. 建立数据截止时间约束。
4. 建立可信度和风险标记。
5. 防止Aurora直接输出交易建议。
6. 测试多个行业和公司。

## 任务量

- 接口设计：3～5天
- 适配开发：5～8天
- 联调：5～10天
- 回归测试：3～5天
- 总量：约15～25人日

## 主要难点

- 应用逻辑与知识底座边界混淆；
- 非结构化观点被误用为量化信号；
- 过时信息进入WQRS；
- 同一事件重复计算；
- 可信度不足却影响决策。

## 核查项目

- [ ] Aurora是否只提供研究输入；
- [ ] WQRS是否保留自身风控和判断；
- [ ] 是否有数据截止时间；
- [ ] 是否有来源和证据；
- [ ] 是否有重复事件去重；
- [ ] 是否能禁用低置信输入；
- [ ] 是否有接口版本管理；
- [ ] 是否有联调测试集。

## 阶段产物

```text
sdk/wqrs_client.py
contracts/wqrs_knowledge.schema.json
apps/wqrs_adapter/
```

## 退出条件

WQRS能够稳定调用Aurora数据，但任何Aurora故障不会破坏WQRS核心决策链。

# 十八、当前推荐的下一步

当前不建议直接进入大规模编码。

下一步最合理的顺序是：

```text
Step 1
完成《Aurora核心对象模型与知识标准》

Step 2
选取10～20条真实资料作为测试集

Step 3
人工制作标准答案

Step 4
定义第一个MVP输入与输出

Step 5
再开始搭建代码骨架
```

第一项真正需要完成的文档建议为：

# 《Aurora 核心对象模型与知识标准 V1.0》

这份文档将定义：

- Source；
- Document；
- Fact；
- Claim；
- Evidence；
- Knowledge；
- Insight；
- Personal Opinion；
- Feedback；

以及它们之间如何关联。

它相当于 Aurora 的“数据宪法”。如果这一层没有定稳，后面采集、RAG、Agent和WQRS接入都会反复返工。

# 十九、最终原则

Aurora 的建设顺序必须长期坚持：

```text
先证明价值
再增加功能

先定义对象
再设计数据库

先跑通闭环
再扩大来源

先个人可用
再平台复用

先保留证据
再生成观点

先允许修正
再追求自动化
```

Aurora 不是一次性开发项目。

它是一套需要通过真实使用、持续复盘和长期积累逐步成熟的个人认知基础设施。
